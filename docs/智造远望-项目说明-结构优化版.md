# 智造远望 - 智慧工厂可视化大屏（结构优化版）

## 项目描述
“智造远望”是一套面向制造企业的生产监控与协同管理系统，前端包含 `dashboard` 大屏和 `admin` 后台两端。大屏负责展示产能态势、AGV 运行状态和传感器异常，后台负责设备台账、策略配置和报表分析，整体形成“监控发现问题 -> 定位影响范围 -> 告警确认 -> 后台追踪处理”的业务闭环。

项目的核心价值不在于堆砌可视化组件，而在于把高频实时渲染、低频统计分析和后台支撑能力拆成边界清晰的前端模块，保证大屏流畅性、状态一致性和跨端协同效率。

## 技术栈
- 前端框架：`Vue 3`、`TypeScript`、`Vite`
- 工程体系：`pnpm workspace`、`Turborepo`
- 状态管理：`Pinia`、`@tanstack/vue-query`
- 可视化：`ECharts`、`ZRender`、`AntV S2`
- 后台 UI：`Ant Design Vue`
- 共享层：`packages/shared`、`packages/charts`、`packages/monitor`

## 技术选型

### 1. 工程基建架构（Infrastructure & Build）
项目采用 `pnpm workspace + Turborepo` 的 Monorepo 结构，将 `dashboard`、`admin` 和公共能力包拆分到独立工作区。这样做的目的不是为了“追新”，而是为了把共享类型、网络层、图表能力和监控能力沉淀为稳定的基础设施，避免两端各自维护一套实现。

在构建层面，`Vite` 提供较快的冷启动和开发反馈，`Turbo` 负责工作区级别的增量构建与任务编排。对于大屏这类依赖图表库和渲染引擎的项目，这种拆分方式比单仓多项目复制代码更容易维护，也更适合后续扩展新端能力。

### 2. 应用核心架构（Core Application Architecture）
应用侧采用 `Vue 3 Composition API + <script setup> + TypeScript` 作为统一编码方式。页面层只负责视图装配和交互编排，服务端数据通过 `vue-query` 管理，客户端 UI 状态通过 `Pinia` 或页面级 store 管理。

这里的关键取舍是“状态分治”：
- 低频、可缓存、以接口为准的数据，例如产能概览、传感器趋势、报表查询，交给 `vue-query` 处理缓存、轮询、重试和失效。
- 与界面交互直接相关的状态，例如筛选条件、详情抽屉、聚焦对象、深链参数，交给 `Pinia` 或页面 store。
- 高频实时数据，例如 AGV 坐标和运行态，不进入深响应式链路，而是通过 `Map/DataBuffer` 维护，避免对 Vue 渲染系统造成持续压力。

### 3. 核心视数引擎（Core Visual & Data Engine）
在可视化层，项目没有把所有数据都塞进通用图表组件，而是按场景拆分渲染策略：
- 宏观趋势与统计卡片使用 `ECharts`，适合承载产能趋势、传感器时序和概览指标。
- AGV 主舞台使用 `ZRender + requestAnimationFrame` 做高频渲染，绕过不必要的组件级更新，保证实时性。
- 后台大体量报表场景采用 Canvas 透视表方案，用于承接复杂行列结构和高密度数据展示，避免传统 DOM 表格在大数据量下的滚动卡顿与样式抖动。

技术选型的核心原则不是统一，而是“按场景选择成本最低、收益最高的渲染方式”。

### 4. UI 与视觉体系（UI & Visual Experience）
大屏视觉方向采用工业科技风，重点不是“炫”，而是信息密度和状态表达效率。首页以顶部状态带、中央 AGV 主舞台、右侧情报栏和底部事件轨构成四区布局，把状态总览、异常定位和闭环反馈放在同一视图里。

后台侧采用企业常见的桌面端结构，重点保证列表、详情、策略编辑和报表查询的操作效率，而不是做成一个大而全的运营平台。这样可以让大屏与后台职责边界更清晰，也更符合制造场景下的使用方式。

### 5. 工程化与质量保障（Engineering & Quality）
工程化层面的重点是保证共享层稳定和工作区可持续演进：
- 统一收敛公共导出面，避免业务应用跨包直接引用零散文件。
- 用 `build/test/typecheck` 作为基础质量门禁，优先确保共享包、dashboard 和 admin 在同一工作区内能稳定联动。
- 通过 provider、类型定义和查询封装，降低页面层直接拼接接口响应的复杂度。

对于这类中后台和可视化混合项目，工程质量的价值不在于工具数量，而在于边界是否稳定、改动是否可控、双端是否能一起演进。

## 系统架构设计

### 1. 宏观架构策略（Architecture Strategy）
系统整体基于 Monorepo 构建，采用“渲染链路”和“业务管控链路”分离的设计思路：
- `dashboard` 负责实时态势感知与异常展示，强调实时性和低干扰。
- `admin` 负责设备信息、策略配置和报表查询，强调可追踪和可维护。
- `packages/shared` 承担类型、数据提供器、网络查询和跨端通信能力。
- `packages/charts` 承担图表与渲染层能力。
- `packages/monitor` 承担端侧异常与性能观测能力。

这种拆分方式可以把高频渲染问题与后台管理问题隔离开，同时又保留跨端共享能力，避免工程在演进过程中逐渐失控。

### 2. 工程物理目录拓扑（Directory Topology）
项目目录可以概括为下面这套结构：

```text
smart-manufacturing-mono/
├── apps/
│   ├── dashboard/            # 工业监控驾驶舱
│   └── admin/                # 闭环支撑后台
├── packages/
│   ├── shared/               # provider、query、types、websocket
│   ├── charts/               # zrender / echarts 渲染能力
│   ├── monitor/              # 端侧监控与异常采集
│   └── config/               # TS / Vite 等基础配置
├── docs/                     # TDD 文档与实施记录
├── turbo.json
└── pnpm-workspace.yaml
```

这样的目录结构适合在面试里说明两件事：一是应用层和基础设施层有清晰分界，二是共享能力不是零散 utils，而是可以独立演进的工作区包。

### 3. 核心流转与微架构规范（Micro-Architecture & Data Flow）
系统运行时可以拆成三条互不干扰的数据链路：

**链路一：服务端状态链路**
低频统计数据通过 `vue-query` 获取，例如产能概览、报表查询、传感器趋势。页面只消费聚合后的查询结果，不直接在组件里拼装接口细节。

**链路二：高频渲染链路**
AGV 实时坐标进入 `DataBuffer/Map` 后，由渲染循环按帧读取并更新 `ZRender` 节点。这里的重点是把“数据写入频率”和“视图刷新频率”解耦，避免每次坐标变化都触发 Vue 组件更新。

**链路三：闭环协同链路**
传感器异常或设备状态变化会被聚合为统一的视图模型，大屏展示异常卡片、事件轨和影响范围，用户可以在大屏进行确认，再通过深链跳转后台带入 `agvId / sensorId / lineId / alertId` 等上下文，完成后续追踪。

如果要用一句话总结这个架构，可以概括为：低频数据走查询链路，高频数据走渲染链路，闭环动作走上下文链路。

## 项目职责
项目为多人协作，我主要负责以下几块核心能力：

- 负责 dashboard 与 admin 的关键链路落地，推动大屏监控与后台支撑从“展示型页面”收敛成可闭环的业务流。
- 负责高频 AGV 渲染链路设计，拆分响应式状态与高频数据缓冲区，保证大屏场景下的流畅性。
- 负责统一数据提供器和共享类型设计，收敛 `auto / api / mock` 三种模式下的数据接入方式。

- 参与后台报表、趋势图和监控能力建设，保证 dashboard 和 admin 在共享能力上的一致性。

## 项目难点与亮点

### 1. 高频实时渲染与响应式边界控制
AGV 坐标属于高频变化数据，如果直接放进深响应式状态，更新频率和节点规模一上来，主线程很容易被拖垮。这里采用 `Map/DataBuffer + requestAnimationFrame + ZRender` 的方式，把高频数据缓冲、节点更新和页面状态管理拆开，只让 Vue 管理需要进入模板的低频控制状态。

这个设计的价值在于：不是简单“绕开框架”，而是在性能敏感场景下主动划清响应式边界。

### 2. 服务端状态与客户端状态分治
项目同时使用 `vue-query` 和 `Pinia`，但两者职责不同。前者负责服务端状态快照，后者只维护客户端交互状态。这样可以减少页面内手写轮询、缓存和重试逻辑，也避免把接口数据、筛选项和弹窗状态混在一个 store 里。

这一点很适合在面试里体现你的架构判断力，因为它解决的是复杂度问题，而不是单纯换一个库。

### 3. 统一 provider 与单一数据源策略
项目的数据提供器支持 `auto | api | mock` 三种模式，但运行时只会选择一个最终来源，不做真实接口与 mock 数据混用。这样可以同时满足演示环境、联调环境和正式接口接入阶段的需求，又不会让页面逻辑被分叉条件污染。

这部分亮点不在于“模式多”，而在于 provider 让页面消费的是统一视图模型，而不是零散接口结果。

### 4. 大体量趋势图的数据表达效率
传感器时序和产能趋势图的数据量一旦拉长到月级别，图表的瓶颈通常不在接口，而在浏览器端渲染。项目在趋势图场景里采用了大数据量配置和降采样思路，让图表优先保留趋势和异常峰值，避免把大量无效重叠点直接压给浏览器。

对外讲解时，这部分重点要落在“为什么前端仍然需要做数据表达优化”，而不是只说自己会调图表参数。

### 5. 后台大体量报表与透视分析场景
后台报表不是普通表单页，它承接的是多维度、跨产线、跨班次的数据浏览需求。这里采用 Canvas 透视表方案，目的不是炫技，而是为了在复杂表头和大数据量下保持更稳定的滚动与交互体验，同时为后续多维分析留出空间。

如果你在面试里保留这块内容，建议聚焦“为什么普通 DOM 表格不适合这个场景”。

### 6. 监控与稳定性兜底
对于大屏这类长时间运行的前端系统，稳定性不是附加项。项目通过端侧监控能力补充异常采集、性能指标和网络链路观察，用来支撑现场问题排查。这里可以讲清楚：为什么大屏系统比普通后台更需要监控，以及为什么监控要贴近具体业务链路。

## 高频面试问答（优化版）

### 1. 为什么 AGV 渲染没有直接走 Vue 响应式？
因为高频坐标更新和组件渲染更新不是一个量级的问题。Vue 很适合处理筛选条件、弹窗、详情抽屉这类交互状态，但不适合承担每秒多次、成百上千节点的坐标更新。这里把高频数据放进 `Map/DataBuffer`，再由 `requestAnimationFrame` 驱动 `ZRender` 更新，是为了把数据写入压力和组件更新压力拆开。

### 2. 为什么同时使用 Vue Query 和 Pinia？
因为它们解决的是两类不同问题。`vue-query` 负责服务端状态快照，例如缓存、轮询、重试和失效；`Pinia` 负责界面交互状态，例如当前筛选项、焦点对象和抽屉显隐。这样分层以后，页面代码会更稳定，也更容易解释状态边界。

### 3. 为什么要做统一 provider，而不是页面里自己判断用 mock 还是 api？
因为页面应该只消费统一的数据模型，而不应该感知底层数据来源。provider 的价值在于把 mock、联调和真实接口的切换收敛到一层，页面层只依赖稳定的类型和聚合视图，这样更利于演示、联调和后期替换。

### 4. 为什么这个项目适合用 Monorepo？
因为 dashboard 和 admin 在展示层差异很大，但底层类型、数据模型、网络查询和监控能力高度重合。Monorepo 可以把这些基础设施抽到共享包里统一维护，既避免复制代码，也降低双端口径不一致的问题。

### 5. 高频渲染链路怎么避免内存泄漏？
关键是明确生命周期边界。渲染实例、动画循环和数据缓冲区都要在页面卸载时显式清理，尤其是 `cancelAnimationFrame`、渲染实例销毁和缓冲区清空，不能只依赖框架自动回收。

### 6. 后台大报表为什么不用普通虚拟列表？
虚拟列表适合一维长列表，但复杂报表往往还包含多级表头、横向滚动和多维交叉。这个场景的瓶颈不只是节点数量，还包括布局计算和滚动过程中的重绘成本，因此更适合采用 Canvas 透视表方案。
